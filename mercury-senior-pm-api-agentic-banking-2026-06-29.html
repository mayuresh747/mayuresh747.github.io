The bet

Mercury's API works, but it is mostly read-only and gated. The unlock is not more endpoints. It is a trust layer that lets a third-party AI agent safely move money on a customer's behalf.

AI agents can look at an account through the read-only MCP server but cannot move money. Write actions sit behind a static IP allowlist, and third-party OAuth needs manual approval (Mercury MCP docs). I would build an agent authorization model, expose it first through the API and CLI, and only later through the MCP. Get the trust layer right and Mercury owns the part of the agent economy Stripe and Visa cannot: agent-to-bank, the source of funds.

API
The programmatic doorway that lets software talk to Mercury instead of a person clicking buttons.
MCP (Model Context Protocol)
A standard way for AI tools like Claude and ChatGPT to connect to a service.
CLI
A command-line tool developers run in a terminal.
Write action
Anything that changes state, like sending a payment, versus a read that only looks.

The problem

Today the API is described in the role itself as "niche read-heavy usage." The reason is structural, not a lack of interest. Three groups hurt: developers who want to automate their own accounts, software vendors who want to build banking into their product for many Mercury customers at once, and AI agents that can read an account but cannot act on it.

Why write usage stays niche

  • The MCP server is read-only and in beta. An agent can see balances and transactions, but it cannot pay an invoice or issue a card (OpenBanking Tracker).
  • The write API requires a static IP allowlist. Cloud-native apps and agent runtimes get a different IP every run, so they are locked out by design.
  • OAuth approval is manual and case-by-case. That throttles the third-party developer ecosystem at the door.

The caution is right, but it is now the ceiling

Mercury's caution is correct for building trust early. Command, its new AI assistant, routes every money movement through a confirmation step and never passes card numbers or SSNs to the model (Mercury blog). That is the right conservative baseline. But human-in-the-loop confirmation on a first-party assistant does not give an outside developer or an autonomous agent a safe path to act.

What it costs the business

While Mercury stays read-only, competitors chasing the same startup customers are shipping write-capable agentic surfaces. Ramp's AI assistant grew from 275 to 2,500-plus businesses in months, and 62% of that usage is completion work like coding transactions and approving requests, not just queries (Ramp blog). Brex shipped a two-way ERP-sync accounting API in January 2026. If a technical founder can do more through a competitor's API, developer-led acquisition leaks away for the same audience Mercury already serves.

Who it is for

A job to be done describes the job a user hires a product to do, in the form: when [situation], I want to [motivation], so I can [outcome].

Primary user: the builder When I am running my company's finances in code or with an AI agent, I want to let my software move money and manage accounts safely, so I can stop doing banking operations by hand.
Secondary user: the third-party software vendor When I build banking features into my product, I want one integration that works across all my Mercury-using customers without a manual approval per customer, so I can ship and scale.

The buyer is not always the user

At a startup, the founder or finance owner approves connecting an agent and sets the limits. They carry the risk if an agent sends a wrong payment. So the emotional job matters as much as the functional one: the buyer needs to feel in control and able to prove, after the fact, exactly what every agent did and under whose authority. The functional job is automation. The job that drives adoption is trust.

This maps directly to Mercury's own design choice in Command: permissions, approval rules, daily limits, and spend controls all carry over automatically (Mercury blog). The platform version of that is the product to build.

Approach

Three directions. I would run cheap tests before betting the roadmap on any one.

Hypothesis 1: the blocker is authorization, not endpoints

Write usage is niche because there is no safe model for delegated, revocable, auditable agent action, not because of a missing endpoint. Test: interview the developers and vendors already pushing against the IP allowlist and OAuth queue. If their asks cluster on "let my agent act safely and let me prove what it did," authorization is the wedge.

Hypothesis 2: Mercury's wedge is agent-to-bank, not agent-to-merchant

Stripe and Visa are building rails for agents to pay merchants (Stripe newsroom, Visa Intelligent Commerce). Mercury sits on the other side, the source of funds. Owning the bank-side authorization (which agent, on whose authority, within what limits) is defensible and distinct. Test: map where Mercury's data and money movement plug into the emerging agent protocols, and decide which to support rather than defaulting to a pass-through provider owning the connection.

Hypothesis 3: phase the roadmap on the bank charter

Mercury has OCC conditional approval for a national bank charter, with FDIC and Federal Reserve sign-off still pending (Banking Dive). What the API can expose changes between today's partner-bank setup and a future where Mercury runs its own rails. Test: build the roadmap in two phases so the team is not blocked on regulatory timing, and so charter capabilities slot in when they land.

How I would decide: hypothesis 1 is testable in weeks through customer conversations and the existing OAuth and allowlist backlog. It gates the rest, so I would validate it first.

What I'd ship first

A scoped, revocable, fully-audited agent payment, run by a small set of design partners through the API and CLI, not the MCP yet.

The biggest assumption is that an outside agent can be allowed to move money safely. The cheapest honest test is to let a few trusted developers do exactly that, with hard limits and a complete record, and watch what breaks. The first slice would include:

  • A scoped agent token. An identity that is clearly an agent, tied to a human who authorized it, with a spend limit and an expiry the customer sets.
  • A confirmation policy the customer chooses. Auto-approve under a threshold, human confirmation above it. This reuses the carry-over-permissions model Command already established.
  • A full audit trail. Every agent action recorded with which agent, whose authority, what limit, and the outcome, so the buyer can prove what happened.
  • One real write path. ACH or wire payment initiation, since payments are the action with the clearest demand and the clearest risk.

What it deliberately leaves out, and why

  • No MCP write access yet. Low confidence until the token and audit model is proven on the API path.
  • No card issuance or other write actions in v1. Payments first, by RICE.
  • No self-serve at large scale yet. Design partners first, so failures are contained and observed before the open ecosystem gets the keys.

Metrics

The North Star is the single number that best proxies real value: it should rise only when a customer genuinely won, move before revenue, and be something the team can influence.

North Star

Weekly programmatic write actions successfully completed by authorized agents and integrations. Real money movement and account changes, not read queries. It rises only when builders trust the platform enough to let software act, which is the whole bet.

Guardrail and anti-metrics

The fraud or erroneous-payment rate on agent-initiated actions must stay at or below the human-initiated baseline, or the whole platform loses trust. I would also watch unauthorized or over-limit attempts that the system correctly blocked, as a sign the controls work. The anti-metric to refuse to chase: raw API call volume. More read calls can look like growth while the read-heavy problem gets worse. The point is authorized write actions, not traffic.

30 / 60 / 90

Days 0 to 30

Listen and find the real constraints

  • Interview the developers and vendors already blocked by the IP allowlist and the OAuth approval queue. What can they not build, and why.
  • Pull the real usage mix: how many of the 300K customers use the API, read versus write, and how big the OAuth backlog is.
  • Sit with engineering on the authorization architecture and the deterministic-execution boundary already built for Command.
  • Map Mercury's position against the agent protocols (Trusted Agent Protocol, Machine Payments Protocol, and others) so a protocol choice is informed, not defaulted.
Days 30 to 60

A quick win and a sharpened problem

  • Ship a fast, safe win that does not need the full trust layer: self-serve OAuth for a defined set of read scopes, removing the manual-approval bottleneck for low-risk integrations.
  • Turn the interviews into an evidence-backed problem statement and a two-phase roadmap (partner-bank now, charter-enabled later).
  • Align engineering, growth, partnerships, and risk on the agent authorization model and the MVP scope.
Days 60 to 90

Ship and validate the first bet

  • Launch the scoped agent payment MVP with a small set of design partners, payments only, through the API and CLI.
  • Set the metric baseline: time-to-first-write, agent-initiated error rate, write actions per week.
  • Use partner results to decide the conditions under which write moves to the MCP, and propose the full roadmap.

Risks

Open questions about the proposed work, and how I would test each early.

  • Liability and authorization model. If an authorized agent sends a wrong payment, who carries it, and does the model survive legal and risk review? 65% of financial-services leaders say agentic payments need an entirely new authorization model, and 85% expect agents to execute payments autonomously (Cloud Security Alliance). Validate with risk and legal in week one, before any code.
  • Demand shape. I am assuming the top ask is delegated write, not more read coverage. Validate by interviewing blocked developers in the first 30 days; if the asks cluster elsewhere, reprioritize.
  • Protocol bet. Backing the wrong agent protocol wastes engineering. Validate by mapping where Mercury's source-of-funds role fits each protocol before committing.
  • Charter timing. The roadmap cannot stall on FDIC and Fed approval. Validate the two-phase split so partner-bank work ships now and charter capabilities slot in when approved.
  • Trust regression. Opening write access could raise fraud or errors. Validate by holding the agent-initiated error rate at or below the human baseline as a launch gate, contained to design partners first.

Questions

The first things I would ask the team.

  1. Of the 300K customers, how many use the API today, and what is the read-versus-write split? That sizes the gap precisely.
  2. How large is the OAuth approval backlog, and what is the rejection rate? That tells me how much adoption the manual gate is costing.
  3. What internal red lines (fraud rate, authorization model, liability framework) must be crossed before write actions reach the MCP?
  4. Which agent authorization protocol, if any, has leadership leaned toward, and why?
  5. Will payroll, via the Central acquisition, be exposed through the API, or stay UI-only? That is a major write-surface decision.
  6. After the charter lands, which API capabilities become possible that the current partner-bank setup cannot offer?
