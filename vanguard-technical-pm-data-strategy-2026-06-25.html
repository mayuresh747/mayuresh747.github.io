The bet

The fastest way to grow what Vanguard gets from data and AI is to make the shared facts everything depends on trusted enough that no team has to second-guess them.

The securities, the funds, the counterparties, the codes that classify them: these are the facts every report and model reuses. Vanguard already proved the point. The Virtual Analyst project did not work until the data underneath it was fixed first (AWS ML Blog, April 2026). I would treat each core data set as a product with a named owner and a clear promise, and prioritize the ones that block the most teams.

Master data

The core business records other systems reuse: at an asset manager, securities, funds, clients, counterparties, and legal entities.

Reference data

The standard lists and codes that classify master data: currency codes, country codes, market codes, asset-class categories.

One trusted version of a record

The agreed single answer when systems disagree about the same security or entity. The rule that decides which source wins for each field is what I call a tie-break rule.

A reliability promise

A written commitment on how fresh, complete, and accurate a data set will be, and how fast a problem gets fixed.

The problem

The work is plumbing, but the cost is real money and real risk. When the core record is wrong, the error spreads into every report, trade, and model that touched it.

Who hurts

Portfolio analytics, compliance reporting, pricing, and the fast-growing analyst population all pull from the same core records. Vanguard's analytics platform went from 60 to over 500 analysts and now runs more than 500,000 queries a month (AWS Big Data Blog). Every one of those people inherits whatever quality the shared data has.

What goes wrong

The same security or entity arrives from several sources with conflicting attributes. One vendor says a bond's coupon is 5.0 percent, another says 5.25 percent. The same firm shows up under three names and three identifiers. A record that was right yesterday is wrong today.

Root cause

No agreed single answer, and no maintained tie-break rule for which source wins per field. Corporate actions like splits and mergers need near-real-time updates, so stale records turn wrong fast.

Workarounds, and why they fail

Teams keep private copies and patch them by hand. Those copies drift apart, so two reports built on the same security disagree and no one can say which is right. Manual fixes do not scale to millions of instruments.

What it costs

Across financial services, poor data quality runs about 9.7 million dollars a year for the average firm (IBM, 2025). The bigger cost is the AI agenda: Gartner expects 60 percent of AI projects to be abandoned through 2026 for lack of trustworthy data (Gartner, Feb 2025), and about half of asset managers still lack basic processes to clean and tag their own data (Grant Thornton, Q3 2025). Vanguard is ahead of that average, but the Virtual Analyst lesson says the data layer is the limiter, not the model.

The decision-maker

Ryan Swann's central data office sets the standards, and value is judged by the Value Measurement Office across revenue, cost savings, cost avoidance, and risk reduction (PYMNTS, May 2025). I would write every roadmap item against those four categories, because that is the language the budget is decided in.

Why this problem

I cannot fix every data set at once, so I would rank candidate work with RICE: a score that weighs Reach (how many teams it unblocks), Impact (how much it moves trust and risk per team), Confidence (how sure I am), and Effort (the build cost). High value over low effort wins.

Candidate workReachImpactConf.EffortRead
Securities and instrument master Very high High Med-high High Highest reach, but heavy. Phase it, do not boil the ocean.
Reference data (codes) High High High Low-med Best first move. Small surface, breaks a lot when wrong, fast to fix.
Counterparty / legal-entity resolution Medium High Medium Very high Real value, but matching the same firm across names is slow. Later.
Fund master Medium Medium Medium Medium Important, but narrower blast radius than securities or codes.

My read: start where a small fix unblocks the most downstream work. Reference data is the cheapest lever with wide reach, because a single bad code spawns exceptions in every transaction that touches it (Gresham Tech). I would pair that with the most-used slice of the securities master, not the whole thing. I would not start with counterparty matching: the value is real but the effort is highest and the confidence lowest, so it is a worse first bet.

Approach

The shape of the fix is well understood; the choice is which pattern fits which data set. I would run three directions and decide per domain with data, not dogma.

  • A. Consolidation. Pull all sources together, apply the tie-break rules, publish one agreed copy downstream. Best when pricing, accounting, and compliance all need the same version at the same time. My default for securities and reference data.
  • B. Registry. Keep a cross-reference index that points to where each value lives, without moving the data. Lower lift, good for entity matching where the data is too varied to fully merge yet.
  • C. Coexistence. Maintain one master copy but let domains keep local copies inside agreed drift limits. Useful where a team needs speed and can tolerate a small, bounded lag.

How I would decide: profile the actual data first with SQL, count the conflicts and how often sources disagree, then pick the pattern that matches the real mess. I would publish each result the way Vanguard already does it, into the internal Marketplace through the Data Product Fast Lane, so I am extending the existing machine, not building a parallel one (Vanguard Tech, Jan 2025).

What I'd ship first

The smallest test of the biggest assumption. That one well-governed reference data set, with a named owner and a written reliability promise, measurably cuts downstream errors and gets adopted. I would prove it on one high-traffic reference data set plus its most-used securities fields, end to end, before scaling.

The first slice would include:

  • A profile of the current data using SQL: how many conflicts, how stale, how many private copies exist today.
  • One agreed copy with documented tie-break rules and the source-of-value recorded for each field, so anyone can see where a value came from.
  • Automated quality checks on freshness, completeness, duplicates, and validity, wired into the existing pipeline checks.
  • A written reliability promise and a published entry in the Marketplace, so downstream teams can find it and depend on it.

Why this first, in RICE terms: highest reach for the effort, high confidence because Vanguard's own platform already runs this pattern at scale, and low enough effort to ship inside a quarter. What it deliberately leaves out: full counterparty matching, the complete securities master, and any new tooling purchase. Those wait until the pattern is proven on one domain.

Metrics

North Star: trusted records actually used downstream. A perfect data set no one uses is a failure in the Marketplace model, so I tie quality to adoption in one number.

Inputs that move it: the four quality dimensions the JD names (accuracy, completeness, consistency, timeliness); time from data need to usable data, which Vanguard measured as going from days to minutes on the Virtual Analyst; the count of teams consuming the governed copy; and reliability, meaning promises met and defects caught before they reach a report.

Guardrails, so I do not win the wrong way

  • No new delay introduced into pricing or trading pipelines. Better data is worthless if it arrives late.
  • The number of private duplicate copies should fall, not rise. If teams keep going around the governed copy, I have not earned their trust yet.

Business impact

Vanguard reported over 300 million dollars in incremental value from data and AI in a single year, and judges each initiative across revenue, cost savings, cost avoidance, and risk reduction (PYMNTS, May 2025). I would frame the master and reference data roadmap against those same four categories, as hypotheses to validate, not promises:

Value categoryHow this work would move it (hypothesis)
Risk reductionFewer reference data errors means fewer compliance and reporting breaks, which is the most expensive failure mode in financial services.
Cost avoidanceA trusted shared copy removes the duplicated manual cleanup each team does today on its private copy.
Cost savingsCatching defects before production cuts the hours spent on incident triage and rework.
Revenue generationReliable core data is the prerequisite that unblocks AI use cases like the Virtual Analyst, which the firm is scaling across business units.

I would baseline each of these in the first 90 days so the roadmap is argued in numbers, not adjectives.

30 / 60 / 90

Days 0 to 30

Listen and profile

Meet the data engineers, stewards, and the loudest downstream consumers. Sit in on pipeline standups to see where failures cluster. Profile two or three candidate data sets with SQL. Read the existing Marketplace standards so I extend them, not reinvent them.

Output: a ranked, evidence-backed list of which data sets hurt the most.

Days 30 to 60

A quick win and alignment

Pick the single reference data set with the worst pain-to-effort ratio. Fix it end to end: agreed copy, tie-break rules, quality checks, a written reliability promise, a Marketplace entry. Align the central data office and one business domain on a shared definition of done. Set the baseline metrics.

Output: one governed data set live, with before-and-after numbers.

Days 60 to 90

Validate and propose

Confirm the quick win reduced downstream errors and got adopted by at least one or two teams. Use that proof to propose the phased roadmap for the securities master and the next reference domains, each item mapped to a value category.

Output: a validated pattern and a funded roadmap.

Questions

  1. Which master data domains does this role own first: securities, funds, client, counterparty, or a defined subset?
  2. Does this role sit inside the central data office, or co-located with a business domain under the hub-and-spoke model? That changes who I align with first.
  3. How mature is the Marketplace today: how many data sets are published, and how many fully meet the foundational, technical, and governance standards?
  4. Which reference data vendors feed securities and market data, and where do those feeds land before they reach downstream systems?
  5. Is there an existing tool for managing the trusted copy, or is the approach custom on AWS, as the Fast Lane blog suggests?
  6. How does the Value Measurement Office want a data-quality roadmap item quantified, so I frame it the way the budget is decided?
